Agents Need a Real R Session

mcp-repl, and the limits of shell-only R access

The first move starts simply

Rscript -e is a reasonable first move.

One shell call, one immediate result.

Timing: ~0:25.

Even for a larger prompt, this is a natural place to start. The model calls the shell tool and runs an inline expression with Rscript -e. That is portable, easy to generate, and often good enough for the first look at the data.

Then the request has another step

  • The expression grows.
  • Setup needs to be reused.
  • The next thought needs somewhere to go.

Timing: ~0:25.

But a typical analysis has more than one step. The next move is also natural: write a temporary script, run it, and read the output. This is less brittle than one large shell string, and it gives the agent somewhere to put setup, intermediate objects, and the next piece of the analysis.

Then iteration keeps that pattern

run 1: load -> inspect
run 2: load -> clean -> plot
run 3: load -> clean -> fit
run 4: load -> clean -> debug

The chat remembers.

R starts over.

Timing: ~0:30.

As you ask the model to iterate, explore, and correct itself, it keeps going in the same direction: edit the file, run the file, inspect the output, edit the file again. From the user’s point of view, the conversation is continuous. From R’s point of view, each attempt is a fresh process. The pattern has started to expose the mismatch.

The problem is the shape

The chat has continuity

The request, plan, corrections, and intermediate reasoning are all still in view.

The R process does not

Objects disappear after each run, so the agent rebuilds the world from files.

We are asking an interactive language to behave like a batch job.

Timing: ~0:25.

This is the core mismatch. The agent is not failing because it is lazy or careless. It is adapting to the interface we gave it. If the only primitive is command execution, the agent will build a workflow out of command execution. But R users already know that the richest work in R is usually not a sequence of isolated batch jobs.

R already has the surfaces agents need

Help

?lm

help("predict")

vignette()

Graphics

plot(fit)

ggplot(df, aes(...))

Debugging

browser()

Browse[1]> n

where

EDA

df, fit, res

str(), ls(), glimpse()

These are not just commands. They are a conversation with a live R session.

Timing: ~0:35.

R has always had rich interactive surfaces: objects that stay in memory, help pages, vignettes, graphics devices, readline prompts, and the debugger. Many of these assume a live session. Some need interactive() to be true. Batch execution can approximate parts of this by writing more files and replaying more setup, but it is an approximation.

The missing primitive is a REPL

Agent
↔︎
repl(“…”)
↔︎
Live R session

State persists

Create an object once. Inspect it later.

Outputs return

Text, images, prompts, errors, and warnings come back through one channel.

Debugging works

The agent can step, inspect, and recover inside the same session.

Control is explicit

Interrupt when stuck. Reset when a clean slate is better.

Timing: ~0:30.

The missing primitive is not a smarter shell command. It is a persistent REPL: a live R session the agent can use for the length of the task. The agent should be able to create a model in one call, inspect it in the next, draw a plot, ask for help, step through browser(), interrupt a bad computation, and reset when needed.

mcp-repl is the small interface around that idea

MCP server

Runs over stdio, so external agents can attach without a bespoke integration.

repl

Send input to the live worker and collect the result.

repl_reset

Discard session state and start clean.

Worker runtime

Long-lived R or Python process with captured text, images, prompts, and debugger interaction.

The public surface is small. The important property is continuity.

Timing: ~0:30.

mcp-repl is one implementation of this interface. The MCP tool surface is intentionally small: send input to the REPL, or reset the session. Most of the work happens below that surface: worker lifecycle, output capture, image capture, timeouts, interrupts, prompts, and final response handling. The effect for the agent is simple: R remains there between turns.

Keep the session in a box

Agent-owned R session
  - private runtime state
  - explicit interrupt
  - explicit reset

Sandbox
  - workspace and temp writes only
  - network off unless configured
  - recovery path when the run goes wrong

Persistence is useful only if the boundary is clear.

Timing: ~0:25.

This is not the same as a human and a model sharing one IDE console. For agent work, the live session needs a boundary. The model can keep objects in memory, but writes are constrained, network access is off unless configured, and the host has explicit controls for interrupt and reset. The sandbox is part of what makes a persistent session acceptable.

This matters when no one is sitting there

Unattended work

  • fresh-data reconnaissance
  • scheduled checks
  • overnight failure analysis
  • first-draft reports

Workflow evaluation

  • realistic R tasks
  • debugger and help use
  • recovery after errors
  • regression tests for tool behavior

A real session makes the real workflow observable.

Timing: ~0:35.

The same primitive matters beyond a live demo. A human may review the result, but the session itself can be owned by the agent while work is happening: scheduled checks, fresh-data reconnaissance, overnight analysis, or report drafting. It also gives us a better way to evaluate agents. We can test whether they can inspect state, use help, view plots, step through the debugger, and recover after mistakes.

Two useful shapes

Timing: ~0:35.

These are related, but they are not the same product shape. mcp-repl is agent-facing. It gives external agents a private R session that can run without a human directly attached to the console. Posit Assistant is the richer interactive path inside the IDE, where the human, the assistant, and the R session are all part of the same environment.

Takeaway

If an agent is going to work in R, it should get a live R session, not only a shell.

mcp-repl is the open-source, agent-facing path: persistent, interactive, private, and sandboxed.

For human-in-the-loop interactive analysis, the integrated IDE path is Posit Assistant.

https://github.com/posit-dev/mcp-repl

Timing: ~0:20.

The takeaway is narrow. R support for agents should not stop at command execution. For agent-owned sessions, mcp-repl gives R a persistent interactive surface over MCP. For interactive work with a human in the loop, Posit Assistant is the integrated IDE path.
